\# UUID Strategy



\## Purpose



The Collectible Authentication Platform uses Universally Unique Identifiers (UUIDs) as the primary identifier for every database entity.



Using UUIDs provides globally unique identifiers that are independent of database implementation, simplifies distributed systems, and prevents predictable record identifiers from being exposed externally.



All primary keys and foreign keys throughout the platform follow this strategy.



\---



\# Primary Keys



Every table uses a UUID as its primary key.



The primary key column is always named:



```text

guid

```



Current entities:



```text

title



media\_item



collection



listing



authentication\_log



audit\_log

```



Each table contains:



```text

guid (PK)

```



\---



\# Foreign Keys



Foreign keys reference the UUID of another entity.



Foreign key naming follows the convention:



```text

<referenced\_table>\_guid

```



Examples:



```text

title\_guid



media\_item\_guid



collection\_guid



listing\_guid



user\_guid

```



Foreign keys always reference the `guid` column of the parent entity.



\---



\# UUID Generation



UUIDs are generated by the application before records are inserted into the database.



Once assigned, a UUID never changes.



UUID generation is independent of Cloudflare D1 and remains consistent across all application components.



\---



\# Immutability



UUIDs are immutable.



A UUID must never be:



\- Modified

\- Reused

\- Reassigned

\- Recycled



Once created, the UUID permanently identifies the record for its entire lifecycle.



\---



\# External Identifiers



UUIDs are the only identifiers exposed outside the application.



Examples include:



\- NFC authentication

\- API requests

\- Marketplace integrations

\- Internal application references



Sequential database identifiers should never be exposed externally.



\---



\# Referential Integrity



Relationships between tables are maintained using UUID-based foreign keys.



Examples:



```text

title

&#x20;   guid

&#x20;       │

&#x20;       ▼

media\_item

&#x20;   title\_guid



collection

&#x20;   media\_item\_guid



listing

&#x20;   collection\_guid



authentication\_log

&#x20;   collection\_guid

```



This ensures every relationship remains globally unique and independent of database implementation.



\---



\# Benefits



Using UUIDs provides several advantages:



\- Globally unique identifiers

\- Database independence

\- Easier distributed processing

\- Safer public identifiers

\- Better support for synchronization

\- Simplified data migration

\- Improved scalability



\---



\# Future Compatibility



The UUID strategy is intended to remain unchanged as the platform evolves.



Future entities such as:



\- User

\- Marketplace

\- NFT

\- API Key

\- Marketplace Sync



will follow the same conventions and use UUID primary keys with UUID-based foreign key relationships.



This provides a consistent identification strategy across the entire platform.
